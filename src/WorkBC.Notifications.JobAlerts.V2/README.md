# WorkBC Job Alert Notifier (PHP)

PHP 8.3 rewrite of the legacy **WorkBC.Notifications.JobAlerts** .NET console
app. Runs as the `jb-notifications` Kubernetes CronJob (and inside the
`php-cli` / cli2 operator shell) and:

1. decides which alert frequencies are due today
   (daily always; weekly on Mondays; biweekly on the 15th and last day of the
   month; monthly on the 1st),
2. loads the matching `JobAlerts` rows (active job seekers, not deleted) from
   PostgreSQL,
3. re-runs each alert's saved `JobSearchFilters` JSON against Elasticsearch
   with the alert-frequency date window applied (count-only query), and
4. emails every job seeker whose alert matched at least one new job, using the
   `email.jobAlert.*` templates from the `SystemSettings` table.

## Running

```bash
composer install
cp .env.example .env       # fill in DB / ES / email settings
php src/notify.php
```

Emails are only sent when `IS_PRODUCTION=true` **or** `SEND_EMAIL_TESTING_TO`
is set (in which case every email is redirected to that address) — the same
safety gate the .NET task had.

Inside the running Linux stack:

```bash
docker-compose -f docker-compose.yml -f docker-compose.linux-dev.yml \
    run --rm notifications-jobalerts-v2
# or from the php-cli shell:
cd /app/workbc-notifications-jobalerts-v2 && php src/notify.php
```

## Parity with the C# implementation

The port reproduces the legacy output byte-for-byte, including its quirks
(uppercase `"DESC"` sort directions, the DatePosted range without a
`time_zone`, trailing-comma cleanup, `C#` → `C` keyword sanitization). Don't
"fix" these unilaterally — the .NET search API (`WorkBC.ElasticSearch.Search`)
still builds the same queries for the live job search, and the two must stay
in sync.

`tests/golden/` was generated by running `tests/fixtures/` through the real
C# `JobSearchQuery` / `KeywordParsing` / `SalaryRangeHelper` (net6.0, en-CA,
stubbed geocoder). Run the parity suite (no composer install needed):

```bash
php tests/run_tests.php
```

Deliberate deviations from the C#:

- Invalid recipient addresses are skipped silently before calling the mail
  transport (the C# only skipped them via `FormatException` on the SMTP path;
  the intent — not spamming the logs with ISB-cleansed `xxxx` addresses —
  applies to every transport).
- Google's geocoding XML is taken from the *first* `locality` component;
  the C# `SingleOrDefault()` would fail the whole geocode if Google ever
  returned two (never observed in practice).

## Configuration

Env vars (see `.env.example`): `DB_*`, `ELASTIC_*`, `INDEX_EN`,
`IS_PRODUCTION`, `SEND_EMAIL_TESTING_TO`, `JB_SEARCH_URL`,
`GOOGLE_MAPS_API_KEY`, `EMAIL_*` (SES → SMTP → SendGrid selection order),
`PROXY_*`. The mapping from the old .NET appsettings keys is documented in
`src/Config/AppConfig.php`.
